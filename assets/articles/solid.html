Let's talk <SOLID>!

Sept 19, 2023

The SOLID principles are a set of five design principles in object-oriented programming and software development that aim to create robust, maintainable, and scalable software systems.


Four Pillars of OOP


    The SOLID acronym stands for:


    Single Responsibility Principle(SRP):

    • States that a CLASS should have only ONE REASON to CHANGE.
    • This means that each class should have only one responsibility or job to perform, and that responsibility should be encapsulated within the class.

    Open / Closed Principle(OCP):

    • States that software components(classes and methods) should be OPEN for EXTENSION and CLOSED for MODIFICATION.
    • Means you have the ability to add new features or functionality to a software system without having to modify the existing code.

    Liskov Substitution Principle(LSP):

    • States that subtypes must be substitutable for their base types without causing any unexpected behavior.
    • This means that if a program is designed to work with a certain type, it should also work correctly with any subtypes of that type.

    Interface Segregation Principle (ISP):

    • States that clients should not be forced to depend on interfaces they do not use.
    • This means that you should design interfaces that are specific to the needs of the clients that use them, rather than creating interfaces that contain methods that are not relevant to some clients.

    Dependency Inversion Principle(DIP):

    • States that high-level modules should not depend on low-level modules; both should depend on abstractions. Abstractions should not depend on details; details should not depend on abstractions.
    • This means that you should design your code in a way that high-level modules are not tightly coupled to low-level modules, and that dependencies are inverted so that abstractions are used to decouple components.


Components


  1. Model: The Model represents the application's data and business logic. It is responsible for managing and manipulating data, as well as responding to requests for information. In many cases, the Model is also responsible for data validation and persistence. It ensures the integrity of the data and communicates any changes to the View and Controller.

  2. View: The View is responsible for presenting the data to the user and rendering the user interface. It takes data from the Model and displays it in a format that is understandable to the user. Views can include various elements like buttons, text fields, images, and more. They are typically passive and do not contain application logic.

  3. Controller: The Controller acts as an intermediary between the Model and the View. It handles user input and translates it into actions that affect the Model or View. Controllers receive user requests, process them, update the Model as necessary, and then update the View to reflect any changes. They contain the application's logic for responding to user interactions.
